**src/diagram/annotations.ts:** Implement the annotation parsing system. Follow the research code examples closely:

Constants:
- `ANNOTATION_START = '%% --- ANNOTATIONS (auto-managed by SmartB Diagrams) ---'`
- `ANNOTATION_END = '%% --- END ANNOTATIONS ---'`
- `FLAG_REGEX = /^%%\s*@flag\s+(\S+)\s+"([^"]*)"$/`

Functions:
1. `parseFlags(content: string): Map<string, Flag>` -- Parse all `%% @flag` lines from within the annotation block. Skip unrecognized lines tolerantly (log a debug warning via stderr logger, do not throw). Return a Map keyed by nodeId.
2. `stripAnnotations(content: string): string` -- Remove the entire annotation block (from ANNOTATION_START to ANNOTATION_END inclusive) and any trailing blank lines. Return pure Mermaid content.
3. `injectAnnotations(content: string, flags: Map<string, Flag>): string` -- Strip existing annotations first, then append a new annotation block at the end. Escape double quotes in flag messages by replacing `"` with `''`. If flags map is empty, return content with no annotation block.

Import `Flag` from `./types.js`. Import `log` from `../utils/logger.js`.

**src/diagram/parser.ts:** Implement basic .mmd content parsing:

1. `parseDiagramType(content: string): string | undefined` -- Extract the diagram type from the first non-empty, non-comment line. Match against known types: `flowchart`, `graph`, `sequenceDiagram`, `classDiagram`, `stateDiagram`, `erDiagram`, `gantt`, `pie`, `gitgraph`, `mindmap`, `timeline`. Return the matched type string or `undefined` if no match.
2. `parseDiagramContent(rawContent: string): { mermaidContent: string; flags: Map<string, Flag>; diagramType?: string }` -- Convenience function that calls `stripAnnotations`, `parseFlags`, and `parseDiagramType`. Returns all three results.

This parser is intentionally lightweight. Detailed AST parsing of Mermaid nodes and edges is not needed in Phase 1 -- the browser-side Mermaid.js library handles rendering. Our parser just separates Mermaid content from our annotations.

**src/diagram/validator.ts:** Implement Mermaid syntax validation:

1. `validateMermaidSyntax(content: string): ValidationResult` -- Try `@mermaid-js/parser` first. The library's API may vary -- investigate the actual exports at implementation time. If `@mermaid-js/parser` works (import `parse` or similar from it), use it to validate. If it throws, extract error message, line, and column info from the error.
2. **Fallback:** If `@mermaid-js/parser` does not work for flowchart validation (it is pre-1.0 and may not cover all diagram types), implement a regex-based heuristic validator that checks:
   - First non-comment line starts with a valid diagram type keyword
   - Basic bracket matching (no unclosed `[`, `(`, `{`)
   - Edge definitions use recognizable arrow syntax
   - Return structured `ValidationResult` with `valid`, `errors[]`, and `diagramType`
3. The validator should be tolerant -- it does not need to catch every possible error. Browser-side Mermaid catches the rest. Focus on catching obvious syntax errors and returning helpful error messages.

Import types from `./types.js`.

**src/diagram/service.ts:** Implement DiagramService class:

```typescript
export class DiagramService {
  constructor(private readonly projectRoot: string) {}
```

Methods:
1. `async readDiagram(filePath: string): Promise<DiagramContent>` -- Resolve path (with traversal protection), read file as utf-8, call `parseDiagramContent` to split Mermaid content from flags, call `validateMermaidSyntax` on the Mermaid content, return full `DiagramContent` object.
2. `async writeDiagram(filePath: string, content: string, flags?: Map<string, Flag>): Promise<void>` -- Resolve path, if flags provided call `injectAnnotations`, create parent directory with `mkdir { recursive: true }`, write file as utf-8.
3. `async getFlags(filePath: string): Promise<Flag[]>` -- Read diagram, return flags as array.
4. `async setFlag(filePath: string, nodeId: string, message: string): Promise<void>` -- Read diagram, update flag in the map, write back with updated flags.
5. `async removeFlag(filePath: string, nodeId: string): Promise<void>` -- Read diagram, delete flag from map, write back.
6. `async validate(filePath: string): Promise<ValidationResult>` -- Read diagram, return validation result.
7. `async listFiles(): Promise<string[]>` -- Use `discoverMmdFiles` on the project root.

Private:
- `resolvePath(filePath: string): string` -- Use `resolveProjectPath` from `../utils/paths.js` to validate and resolve the path. This is the single chokepoint for path security.

All file I/O uses `node:fs/promises`. All imports use `.js` extensions for ESM.

**src/project/manager.ts:** Implement ProjectManager:

```typescript
export class ProjectManager {
  private projects = new Map<string, DiagramService>();
```

Methods:
1. `addProject(rootDir: string): DiagramService` -- Validate rootDir exists (stat it), create a `DiagramService` for it, store in map, return the service. If rootDir already registered, return existing service.
2. `removeProject(rootDir: string): boolean` -- Remove from map, return whether it existed.
3. `getProject(rootDir: string): DiagramService | undefined` -- Lookup by rootDir.
4. `listProjects(): string[]` -- Return all registered project root directories.
5. `async discoverAll(): Promise<Map<string, string[]>>` -- For each registered project, call `listFiles()`, return a map of rootDir to file lists.

The ProjectManager enables DIAG-05 (multi-project support). Each project directory gets its own DiagramService with its own root for path security.

    **Step 1: Download mermaid.min.js**

    Download the pre-built mermaid.min.js (IIFE bundle, single file, no dynamic imports) from the CDN and save as a local media file:

    This avoids the dynamic chunk loading CSP problem described in 07-RESEARCH.md. The file is ~2MB but is the simplest approach.

    **Step 2: Update esbuild.mjs**

    The webview bundle should NOT include mermaid (it's loaded as a separate script tag). Ensure the webview config does NOT bundle mermaid -- it's a global available via the separate script tag. If the webview imports anything from mermaid, mark 'mermaid' as external in the webview esbuild config.

    **Step 3: Implement src/webview/main.ts**

    Full webview script (runs in webview context, bundled as IIFE):

    - Immediately call `const vscode = acquireVsCodeApi()`.
    - Restore state: `const state = vscode.getState() || { currentFile: '', lastContent: '' }`.
    - Type definitions inline (not importing from main project):
      - `WsMessage`: discriminated union matching the server's WsMessage type (file:changed, file:added, file:removed, tree:updated, connected).
      - `ExtensionMessage`: messages from extension host (diagram:update wrapping WsMessage, connection:status).
    - Initialize mermaid (global `mermaid` from mermaid.min.js script tag):
      ```
      mermaid.initialize({
        startOnLoad: false,
        theme: 'dark',
        securityLevel: 'sandbox',
        flowchart: { htmlLabels: true, curve: 'basis' }
      });
      ```
    - `renderDiagram(content: string, file: string)` async function:
      - Call `const { svg } = await mermaid.render('diagram-' + Date.now(), content)`.
      - Clear the diagram container: `container.textContent = ''`.
      - Insert the SVG safely: `container.insertAdjacentHTML('afterbegin', svg)` (mermaid.render returns sanitized SVG -- same pattern used in static/live.html and documented in 07-RESEARCH.md code examples).
      - After rendering, call `initFlagClickHandlers()` from flag-ui module to attach click listeners to nodes.
      - Update state: `vscode.setState({ currentFile: file, lastContent: content })`.
    - Listen for 'message' events on window:
      - `diagram:update`: if msg.data contains a WsMessage of type 'file:changed', extract file and content, call renderDiagram(content, file). Also handle when extension host sends initial diagram data on reconnect.
      - `connection:status`: update the #connection-status element textContent and class.
    - On startup, if `state.lastContent` exists, re-render it to restore the previous view.
    - Import and call `initFlagUI(vscode)` from flag-ui.ts.

    Keep under ~150 lines. The flag UI logic is extracted to flag-ui.ts to respect the 500-line limit and separate concerns.

    **Step 4: Implement src/webview/flag-ui.ts**

    Simplified flag interaction for the sidebar:

    - Export `initFlagUI(vscode: any)`: stores the vscode API reference for postMessage.
    - Export `initFlagClickHandlers()`: called after each render to attach click listeners.
      - Query all `.node` elements in the SVG (Mermaid generates `<g class="node" id="flowchart-NodeId-N">`).
      - For each node, add a click listener that:
        1. Extracts the node ID from the element's id attribute (strip the `flowchart-` prefix and `-N` suffix using regex).
        2. Shows a minimal inline input element positioned near the clicked node.
        3. On Enter/submit: sends `vscode.postMessage({ type: 'addFlag', nodeId, message })`.
        4. On Escape: hides the input.
    - Export `hideFlagInput()`: removes the inline input if visible.
    - The input element is a simple `<input>` absolutely positioned over the diagram, styled with VS Code theme variables.
    - Keep under ~120 lines.

    **Step 5: Update media/webview.css**

    Add styles for:
    - `.flag-input`: position absolute, z-index 100, padding 4px 8px, background var(--vscode-input-background), color var(--vscode-input-foreground), border 1px solid var(--vscode-input-border), border-radius 3px, font-size 12px, min-width 200px
    - `.node:hover`: cursor pointer, opacity 0.8 (visual feedback that nodes are clickable)
    - `.flag-input::placeholder`: color var(--vscode-input-placeholderForeground)
    - SVG overflow visible for proper diagram scaling
    - Dark theme adjustments using VS Code CSS variables

  <action>
    **Update extension.ts** to handle flag actions from the webview:

    1. Add a `handleWebviewMessage(msg: any)` function in activate():
       - If `msg.type === 'addFlag'`: extract nodeId and message. Derive the HTTP base URL from the WS URL (replace `ws://` with `http://`, remove `/ws` path). Read the current file path from a tracked state variable. Construct the flag annotation line `%% @flag ${nodeId} "${message}"`.
       - Use Node.js built-in `http` or `https` module (consistent with project pattern from 06-01 -- avoids fetch experimental warnings) to POST to `${httpBaseUrl}/save` with body `{ filename: currentFile, content: updatedContent }`.
       - To get the current content: the extension host should track the latest content received via WebSocket for each file. When a `file:changed` message arrives, store it in a Map<string, string>. When saving a flag, read current content from this map, append the flag annotation, and POST back.
       - Handle errors: show vscode.window.showErrorMessage on HTTP failure.

    2. Improve the WebSocket message relay in activate():
       - Track `currentFile` (the file currently displayed in the webview) and `fileContents` (Map<string, string> of file -> latest mermaid content).
       - When a `file:changed` WsMessage arrives: update fileContents map, relay to webview.
       - When a `tree:updated` WsMessage arrives: relay to webview (for future file tree support).
       - When the webview sends `{ type: 'selectFile', file: string }`: fetch initial content via HTTP GET to `${httpBaseUrl}/api/diagrams/${file}` and send to webview. Update currentFile.

    3. On webview visible (resolveWebviewView called): if there's a currentFile with cached content, immediately send it to the webview so it doesn't show blank.

    **Update diagram-provider.ts**:

    1. Add `onWebviewMessage` callback property (set by extension.ts) that forwards webview messages to the extension host for handling.
    2. In resolveWebviewView: wire `webview.onDidReceiveMessage` to call `this.onWebviewMessage?.(msg)`.
    3. Add `sendCurrentState(file: string, content: string)` method that posts initial diagram data to the webview on reconnection/reshow.

    Important implementation notes:
    - The flag annotation format must match exactly: `%% @flag nodeId "message"`. See src/diagram/annotations.ts for the format.
    - The annotation block markers are: `%% --- ANNOTATIONS (auto-managed by SmartB Diagrams) ---`. If the file doesn't have this block yet, the flag should be appended as-is. The SmartB server's DiagramService handles annotation block management on the next read cycle.
    - Simplest approach: use the SmartB server's existing infrastructure. Instead of manually constructing annotation lines, use the `/save` endpoint to write the full file content. Read the content from the WS cache, append `\n%% @flag {nodeId} "{message}"`, and POST the result. The DiagramService on the server will re-normalize annotations on the next read cycle.
    - Keep extension.ts under ~150 lines total. If the flag handling logic grows, extract to a separate `flag-handler.ts` module.

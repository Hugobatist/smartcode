  <name>Task 1: Create interaction-state.js — Finite State Machine</name>
  <files>static/interaction-state.js</files>
  <action>
Create `static/interaction-state.js` as an IIFE module (same pattern as all other static/ modules) that provides a lightweight FSM for coordinating all UI interaction modes.

**States (8 total):**
- `idle` — Default. Click selects, right-click opens menu, double-click edits, drag pans.
- `selected` — A node/edge is selected. Click elsewhere deselects, Delete removes, right-click opens menu.
- `editing` — Inline edit overlay is open. All other interactions blocked. Enter/Escape exits.
- `context-menu` — Context menu is open. Click outside or Escape closes.
- `flagging` — Flag mode is active (existing SmartBAnnotations behavior).
- `add-node` — Add node mode (existing MmdEditor behavior).
- `add-edge` — Add edge mode (existing MmdEditor behavior).
- `panning` — Active drag-pan in progress.

**Transition table (object literal, NOT a library):**
```javascript
var TRANSITIONS = {
  idle:           { click_node: 'selected', click_edge: 'selected', right_click: 'context-menu', dbl_click: 'editing', flag_toggle: 'flagging', add_node_toggle: 'add-node', add_edge_toggle: 'add-edge', pan_start: 'panning' },
  selected:       { click_empty: 'idle', click_node: 'selected', click_edge: 'selected', right_click: 'context-menu', dbl_click: 'editing', escape: 'idle', delete_node: 'idle', flag_toggle: 'flagging', pan_start: 'panning' },
  editing:        { confirm: 'idle', cancel: 'idle' },
  'context-menu': { action: 'idle', close: 'idle', edit_action: 'editing' },
  flagging:       { flag_toggle: 'idle', escape: 'idle' },
  'add-node':     { add_node_toggle: 'idle', escape: 'idle' },
  'add-edge':     { add_edge_toggle: 'idle', escape: 'idle' },
  panning:        { pan_end: 'idle', pan_end_selected: 'selected' },
};
```

**Internal state:**
- `currentState` (string) — starts as `'idle'`
- `selectedId` (string|null) — the currently selected node/edge ID
- `selectedType` (string|null) — `'node'` | `'edge'` | `'subgraph'`

**Public API (`window.SmartBInteraction`):**
- `getState()` — returns current state string
- `getSelection()` — returns `{ id: selectedId, type: selectedType }` or `{ id: null, type: null }`
- `transition(event, payload)` — attempts transition. If valid, updates currentState, emits `interaction:transition` event on SmartBEventBus with `{ from, to, event, payload }`. Returns `true` if transition happened, `false` if invalid.
- `select(id, type)` — sets selectedId and selectedType (called by selection.js when selecting)
- `clearSelection()` — sets selectedId and selectedType to null
- `forceState(state)` — force-set state (for existing mode toggles in annotations/editor that need to sync their state with the FSM)
- `isIdle()` — shorthand for `currentState === 'idle'`
- `isBlocking()` — returns true if in `editing` or `context-menu` (states that block all other interactions)

**EventBus emission:** On every successful transition, emit `interaction:transition` with `{ from, to, event, payload }`.

**Target: ~120-150 lines.** Keep it simple — the FSM is a lookup table + event emitter, not a framework.

<task type="auto">
  <name>Task 2: Create selection.js — Selection state, visual indicators, keyboard handlers</name>
  <files>static/selection.js</files>
  <action>
Create `static/selection.js` as an IIFE module that manages node/edge selection, visual SVG indicators, and keyboard shortcuts.

**Dependencies:** `interaction-state.js` (SmartBInteraction), `diagram-dom.js` (DiagramDOM), `event-bus.js` (SmartBEventBus), `diagram-editor.js` (MmdEditor).

**Internal state:**
- `selectedNodeId` (string|null) — mirrors SmartBInteraction.getSelection().id
- `overlayGroup` (SVGGElement|null) — the `<g class="selection-indicator">` element in the SVG

**Core functions:**

1. `selectNode(nodeId)` — Called when a node is clicked in idle/selected state.
   - Clear any existing selection indicator
   - Use `DiagramDOM.findNodeElement(nodeId)` to get the SVG element
   - Get `getBBox()` from the element
   - Create an SVG `<g class="selection-indicator">` group containing:
     - A `<rect>` with: x=bbox.x-4, y=bbox.y-4, width=bbox.width+8, height=bbox.height+8, fill=none, stroke=#6366f1, stroke-width=2, stroke-dasharray=6,3, rx=4
     - Four corner handle `<rect>` elements (8x8px, fill=#6366f1, rx=2) at each corner of the selection rect
   - Append the group to the SVG element (`DiagramDOM.getSVG()`)
   - Update `selectedNodeId` and call `SmartBInteraction.select(nodeId, 'node')`
   - Emit `selection:changed` on EventBus with `{ id: nodeId, type: 'node' }`

2. `selectEdge(edgeId)` — Called when an edge is clicked.
   - Clear existing selection
   - Use `DiagramDOM.findEdgeElement(edgeId)` to get the SVG path
   - Add a CSS class `.selected-edge` to the element (styling: stroke=#6366f1, stroke-width=3)
   - NO corner handles for edges (per research recommendation)
   - Update and emit as above but with type `'edge'`

3. `deselectAll()` — Removes selection indicator SVG group and `.selected-edge` classes.
   - Remove `overlayGroup` from SVG if it exists
   - Remove `.selected-edge` class from any elements
   - Set `selectedNodeId` to null
   - Call `SmartBInteraction.clearSelection()`
   - Emit `selection:changed` with `null`

4. `reapplySelection()` — Called on `diagram:rendered` event.
   - If `selectedNodeId` is set, check if the node still exists in the new SVG via `DiagramDOM.findNodeElement()`
   - If exists: re-call `selectNode()` / `selectEdge()` to re-create the indicator
   - If not exists: call `deselectAll()` and transition FSM to idle

**Click handler (delegated on `#preview-container`):**
- IMPORTANT: Use a combined click handler that checks `e.detail` to disambiguate single vs double click:
  - If `e.detail === 2`: this is a double-click — do NOT select, let inline-edit handle it (transition `dbl_click`)
  - If `e.detail === 1`: set a 200ms timeout. If no second click comes (detail===2), proceed with selection logic.
  - ALTERNATIVE (simpler, per research): Accept that selection fires first on double-click. The FSM will transition `selected -> editing` on the dbl_click event. The brief selection flash is acceptable UX.
- Use the SIMPLER approach: handle `click` directly without timer.
  - If `SmartBInteraction.isBlocking()` return (don't handle clicks in editing/context-menu states)
  - If FSM state is `flagging`, `add-node`, or `add-edge`, return (let existing handlers handle)
  - Skip if target is `.zoom-controls`, `.flag-popover`, `.editor-popover`, `.context-menu`, `.inline-edit-overlay`
  - Use `DiagramDOM.extractNodeId(e.target)` to detect what was clicked
  - If node: `selectNode(nodeInfo.id)`, transition `click_node`
  - If edge: `selectEdge(nodeInfo.id)`, transition `click_edge`
  - If null (empty space): `deselectAll()`, transition `click_empty`

**Keyboard handler (document-level):**
- Check that `e.target` is NOT a `textarea`, `input`, or `[contenteditable]` element
- If FSM state is `selected`:
  - `Delete` or `Backspace`: Call `MmdEditor.doRemoveNode(selectedNodeId)` for nodes; for edges, need to parse edge endpoints. Transition `delete_node`.
  - `Escape`: `deselectAll()`, transition `escape`

**EventBus subscription:**
- Subscribe to `diagram:rendered` → call `reapplySelection()`

**CSS (add to selection.js as a style injection at module load, same pattern as other modules that self-inject styles):**
```css
.selected-edge path { stroke: #6366f1 !important; stroke-width: 3 !important; }
.selected-edge .smartb-edge-path { stroke: #6366f1 !important; stroke-width: 3 !important; }
```
Inject via `document.head.appendChild(styleEl)` at IIFE load time.

**Public API (`window.SmartBSelection`):**
- `selectNode(nodeId)`, `selectEdge(edgeId)`, `deselectAll()`, `reapplySelection()`
- `getSelected()` — returns `{ id, type }` or `null`
- `init()` — attaches click handler to `#preview-container`, keyboard handler to document, EventBus subscription

**Target: ~200-250 lines.** The visual indicator construction is the bulk.

Modify `static/pan-zoom.js` (currently 139 lines) to:

1. **Add movement threshold** to prevent panning on intended clicks:
   - Define `var PAN_THRESHOLD = 3;` (3 pixels of movement before pan actually starts)
   - Add a `var panStarted = false;` state variable
   - In the `mousemove` handler: only start actual panning (applying transform) when the mouse has moved more than PAN_THRESHOLD pixels from the mousedown position. Use `Math.abs(e.clientX - panStartX) > PAN_THRESHOLD || Math.abs(e.clientY - panStartY) > PAN_THRESHOLD` to detect.
   - Set `panStarted = true` once threshold is crossed, so subsequent moves apply transform immediately
   - In `mouseup`: if `!panStarted`, the mouse didn't move enough — this was a click, not a pan. Reset state.

2. **Check FSM state** before starting pan:
   - In the `mousedown` handler, add a check: `if (window.SmartBInteraction && SmartBInteraction.isBlocking()) return;`
   - This prevents panning when in `editing` or `context-menu` states
   - Keep existing checks for `flagMode` and `MmdEditor.getState().mode` for backward compat (they still work without the FSM loaded)

3. **Notify FSM on pan start/end:**
   - When pan actually starts (threshold crossed): `if (window.SmartBInteraction) SmartBInteraction.transition('pan_start');`
   - On mouseup when pan was active: `if (window.SmartBInteraction) SmartBInteraction.transition(SmartBInteraction.getSelection().id ? 'pan_end_selected' : 'pan_end');`

The file should grow from 139 to ~170-180 lines. Well within 500-line limit.
